Tables, results, search and filters

Search vs filter

Guidance from MOJ pattern library.

  • Type a search term and submit the search form
  • See a search results page with filters
  • Users can then filter the search results page further or search again starting from
  • Delta is unique in search and filters are typically present in the same page

Delta hypothesis (and some goals)

  • Our design aspiration is for the user never having to enter a search query; the system will intuitively know what the user wants to see
  • Clicking is better (easier, less prone to error) than typing
  • When user input is required to reduce number of displayed results [free] keyword filtering will be most efficient option in majority of cases

What 'problems' are we looking to solve

  • Complex hidden filters

Extending and optimising for Delta users

Extending UX patterns to better support our users

  • Frequently used filters permanently visible (not hidden behind optional panel)
  • Filter by keyword input box active on page load? Needs testing
  • Where possible make use of full width

Functional requirements

  • Filter and search parameters in URL (to support saving and sharing)
  • Clearly display if and what search and filter parameters exist
  • Sort columns by clicking column header (toggles)
  • Keyword filter box

To table or not to table, that is the question.

  • Tables used to compare data
  • Tables don't handle long data well. Content wrapping on to multiple lines should be avoided
  • If data is long and [scanning comparisons not required then table based results most likely not appropriate. In these instanced divs better: more flexibility and better accessibility options.

Versions / options

  • Results summary box (count, active filter count)
  • Results summary box (count, active filter count)
  • Primary (quick) filters
  • Secondary (advanced) filters
  • Select some / all capability

Misc

  • Search and filter parameters should persists on "user going back" (returning to results page)
